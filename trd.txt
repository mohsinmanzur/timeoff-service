1. Executive Summary
ExampleHR-TimeOff is a purpose-built NestJS microservice responsible for the full lifecycle of employee time-off requests within the ExampleHR platform. It acts as an orchestration layer between employee-facing product surfaces and an authoritative external Human Capital Management (HCM) system (Workday or SAP), which is the single source of truth for leave balances.

The service provides a clean REST API surface for submitting, reviewing, approving, and cancelling time-off requests. It maintains a local SQLite cache of balance data for low-latency reads, while continuously reconciling that cache against the HCM via both real-time per-request lookups and nightly batch ingests.

Key design principles driving this service:
Correctness over availability — when balance data is ambiguous or stale, the system fails safe.
Defensive validation — the service enforces business rules even when the HCM fails to reject invalid payloads.
Idempotent mutations — all state-changing operations are safe to retry.
Auditability — every balance mutation and sync event is durably logged.

🎯 Scope  This document covers the v1.0 architecture, data model, API surface, sync strategy, and engineering trade-offs. It is intended for platform engineers, security reviewers, and integration partners.

2. Problem Statement & Challenges
The following engineering challenges were identified during discovery and architecture review:

Challenge 1 — Stale Balance Cache
External actors (payroll runs, anniversary bonuses, manual HCM adjustments) can modify balances at any time without notifying this service. Our local SQLite cache becomes stale silently. We cannot blindly trust cached values before approving requests that deduct balance.
Challenge 2 — HCM Silent Acceptance of Invalid Writes
The HCM REST API has been observed to return HTTP 200 for balance-adjust calls even when the requested deduction exceeds the employee's actual available balance (e.g., sub-zero net result). This means we cannot rely on HCM error responses as a balance guardrail. The service must independently enforce non-negative post-deduction balance constraints.
Challenge 3 — Composite Identity for Balances
Balances are scoped per employee per location (employeeId + locationId). An employee who transfers locations mid-year accumulates two distinct balance records. All queries, caches, and sync logic must treat the (employeeId, locationId) tuple as the unique key, not employeeId alone.
Challenge 4 — Dual Sync Surface with Conflict Risk
The HCM exposes both a real-time REST API (individual reads/writes) and a batch push endpoint (full corpus). The batch payload represents a point-in-time snapshot. A real-time write that succeeded after the batch snapshot was taken will be overwritten unless the ingest logic reconciles correctly. Last-write-wins without timestamp awareness will cause balance regressions.
Challenge 5 — Idempotency and Double-Submission
Network instability and client retries can cause the same time-off request to be submitted multiple times. Without idempotency controls, this results in duplicate approvals and double deductions. The service must deduplicate at the API boundary using client-supplied idempotency keys.
Challenge 6 — Distributed Approval Race Conditions
Two managers reviewing the same request concurrently (or a request being cancelled simultaneously with approval) can result in conflicting state transitions. Optimistic locking using a version counter on the request row must guard all state machine transitions.
Challenge 7 — HCM Unavailability Degradation
The HCM real-time API has a documented SLA of 99.5%, meaning up to 43 hours of downtime annually. The service must gracefully degrade: serving reads from cache with a staleness header, queuing balance writes, and resuming sync on recovery — without losing any approved deductions.

5. Balance Sync Strategy
5.1 Real-Time Pull (Per-Request)
Before processing any request that reads or mutates a balance, the service fetches the current balance from the HCM real-time API:

GET /employees/{employeeId}/balances?locationId={locationId}
Authorization: Bearer {hcm_service_token}
If-None-Match: {cached_etag}

If HCM returns 200 with new ETag — update cache, proceed.
If HCM returns 304 Not Modified — cache hit confirmed; use cached value.
If HCM returns 5xx or times out — serve from cache, set is_stale=true, attach X-Balance-Stale: true header. Block all balance-deducting operations (approve). Allow read-only operations.

5.2 Nightly Batch Ingest
At 02:00 UTC (configurable) the HCM pushes the full balance corpus to POST /balances/batch-ingest. The service processes this as follows:

Parse and validate each record. Reject malformed rows; log and skip.
For each (employeeId, locationId, leaveType) tuple, compare HCM value with local cache.
Conflict resolution rule: If HCM value timestamp (hcm_as_of) > local synced_at, accept HCM value. Otherwise, retain local value and raise a conflict_flag=1 audit log entry.
Apply pending_days offset: available_days_final = hcm_available - sum(PENDING request days for that key).
Write result to balance_cache in a single transaction per record.
Emit a sync_audit_log entry for every row processed, whether changed or not.

⚠️ Conflict Threshold Alert  If more than 5% of batch rows raise conflict_flag=1, a Slack/PagerDuty alert is fired and the batch is quarantined pending manual review. Automatic application is paused.

5.3 Defensive Balance Validation
Because the HCM does not reliably reject insufficient-balance write attempts, the service enforces the following check before any approval:

effectiveAvailable = hcm_available_days
                   - sum(APPROVED request days, same key, overlapping period)
                   - sum(PENDING  request days, same key)
 
if (effectiveAvailable - daysRequested) < 0:
  throw InsufficientBalanceException()

This check runs after a fresh real-time pull. It is not skipped even if the HCM write later succeeds — the pre-condition is enforced locally.

5.4 Balance Write to HCM
Upon request approval, the service writes the deduction back to the HCM:

POST /employees/{employeeId}/balances/adjust
{
  "locationId":  "LOC-NYC",
  "leaveType":   "ANNUAL",
  "adjustment":  -5.0,
  "referenceId": "req-uuid",
  "reason":      "APPROVED_TIME_OFF"
}

If HCM write succeeds: set hcm_sync_status=SYNCED on the request row.
If HCM write fails: set hcm_sync_status=FAILED; enqueue for retry via exponential backoff (max 5 attempts over 30 min). balance_cache is not updated until HCM confirms.
If all retries fail: mark request as APPROVAL_SYNC_FAILED; alert on-call; human intervention required.

7. Alternatives Considered
Alternative A — No Local Cache (HCM as Exclusive Read Source)
Every balance read goes directly to the HCM real-time API. No SQLite balance_cache table. No sync jobs.

Aspect
Analysis
Pros
Eliminates cache coherence complexity entirely. Always reflects ground truth. Simpler codebase.
Cons (Latency)
Every list query, every approval check incurs an HCM round trip (~200–800 ms). Unacceptable for list endpoints serving dashboards.
Cons (Availability)
Service becomes fully unavailable when HCM is down. No degraded-mode operation possible.
Cons (Rate Limits)
HCM APIs are often rate-limited per tenant. High request volumes trigger throttling.
Decision
Rejected. The cache layer is essential for acceptable latency and graceful degradation.

Alternative B — Event-Driven Sync via Webhook/Message Bus
Instead of polling (real-time pull) and scheduled batch, the HCM pushes balance-change events to a message bus (Kafka / SQS). The service consumes events and updates cache reactively.

Aspect
Analysis
Pros
Near real-time cache freshness without per-request HCM calls. Reduced HCM API coupling.
Cons (Infrastructure)
Requires Kafka/SQS provisioning, dead-letter queue management, and consumer group coordination — significant operational overhead for a v1 service.
Cons (HCM Support)
Not all HCM vendors (particularly on-premise SAP deployments) expose reliable outbound webhooks. Event delivery is not guaranteed without additional retry infrastructure.
Cons (Gap Risk)
Missed events during consumer downtime require a catch-up batch job anyway, negating some simplicity gains.
Decision
Deferred to v2. The dual pull+batch strategy is simpler to implement and operate with the current HCM integration contract. An event-driven layer can be added on top in future without breaking API contracts.

Alternative C — PostgreSQL Instead of SQLite
Use PostgreSQL (or a managed equivalent) as the backing store for production-grade concurrency.

Aspect
Analysis
Pros
Native row-level locking. Mature JSONB for audit payloads. Better concurrent write throughput.
Cons
Requires additional infrastructure (managed DB service, connection pooling). Increases operational surface for a v1 service with modest write volume.
Decision
SQLite with WAL mode and TypeORM is sufficient for v1 projected load (< 500 concurrent employees per node). Migration path to PostgreSQL is preserved by keeping all SQL ANSI-standard and using TypeORM abstractions. A flag in the config will switch the TypeORM datasource.

8. Out of Scope
The following items are explicitly excluded from the v1.0 scope of this service:

Calendar integration for public-holiday-aware day counting (assumed caller-computed).
Multi-step approval workflows (e.g., skip-level approvals, delegation chains).
Email / push notification delivery (a dedicated Notification Service will consume request state events).
Carry-forward balance computation logic (handled by HCM; we only cache the result).
UI or BFF layer. This service is a pure API backend.
GDPR data-subject erasure flows (handled by a separate Data Privacy Service).
Multi-tenancy / multi-company support. All employees share one HCM instance in v1.
Partial-day requests with sub-hour granularity. Minimum unit is 0.5 days.
Real-time WebSocket / SSE push of approval status to clients.
Kubernetes Helm chart authoring and deployment pipeline configuration.
